05 — Mobile Responsiveness

Mobile-First Design & Breakpoints

A large share of parents searching for nanny services do so on their phones. The site must look and work perfectly on all screen sizes — every layout, form, and portfolio card is designed mobile-first and enhanced for larger screens.

📱 Mobile-First Rule: All CSS is written for mobile (320 px+) first. Tablet and desktop styles are added using min-width media queries — never max-width overrides.

Breakpoints

320px

Small Mobile (base styles)

Single-column layout everywhere. Full-width cards, forms, and buttons. Navigation collapses to a hamburger menu. Hero text at 28px. Portfolio cards stack vertically one per row.

640px

Large Mobile / Phablet

Portfolio cards move to a 2-column grid. Form fields can sit side-by-side (e.g. first name + last name). Testimonials show 1 card at a time in the carousel.

768px

Tablet

Navigation expands to a full horizontal bar. Service category cards display in a 3-column row. Portfolio grid moves to 3 columns. Hero gains a side image layout.

1024px

Desktop

Full layout with max-width container (960–1100 px centered). Portfolio grid 4 columns. Testimonials show 2–3 cards side by side. Forms have a two-column field layout.

Mobile-Specific UI Rules

✓

Touch Targets

All buttons, links, and form inputs are minimum 48×48 px to meet WCAG accessibility and mobile usability standards. No tiny tap targets anywhere.

✓

No Horizontal Scroll

overflow-x: hidden on body, and all grids/flex containers use flex-wrap: wrap or responsive grid-template-columns so nothing ever bleeds off-screen.

✓

Readable Font Sizes

Body text minimum 15px on mobile. Input fields set to 16px minimum to prevent iOS Safari from auto-zooming on focus.

✓

Fast Load on Mobile

Images use loading="lazy" and are served in WebP format. No heavy JS frameworks. CSS and JS files are minified before deployment. Target: <3s load on 4G.

✓

File Upload on Mobile

Portfolio photo and Photo ID upload fields use accept="image/*" and capture attribute — allowing users to take a photo directly from their phone camera without needing a file browser.

✓

Sticky "Find a Nanny" Button

On mobile, a fixed sticky CTA button floats at the bottom of the screen on key pages (Home, For Clients) so the primary action is always reachable with a thumb.

06 — SEO Strategy

SEO from the First Deployment

SEO is not an afterthought — it is baked into the HTML foundation before the first git push. Every page is fully crawlable, correctly tagged, and structured for Google from launch day. This gives the site a head-start on organic nanny-service search rankings.

🚨 Must-Have on First Deploy: All items in the "Day-One SEO Checklist" below must be present in the initial GitHub Pages deployment — no exceptions.

Day-One SEO Checklist — Every Page

✓

Unique <title> Tags

Each page has a unique, keyword-rich title. Format: "Page Name | Your Care Crew — Professional Nanny Placement". Max 60 characters.

✓

Meta Description

Unique 150–160 character meta description per page, incorporating the primary keyword. Example for Home: "Find a trusted, vetted nanny for your family. Your Care Crew specializes in professional nanny placement, house helpers, and elder care."

✓

Open Graph & Twitter Card Tags

og:title, og:description, og:image, og:url, og:type, twitter:card on every page — essential for social sharing previews.

✓

Canonical URL

<link rel="canonical"> on every page to prevent duplicate content issues across HTTP/HTTPS or www/non-www variations.

✓

Semantic HTML5 Structure

<header>, <nav>, <main>, <section>, <article>, <footer> used correctly throughout. One <h1> per page, logical h2→h3→h4 hierarchy.

✓

JSON-LD Structured Data

LocalBusiness schema on the homepage and Contact page (business name, address, phone, service types). FAQPage schema on the For Clients page. Helps Google display rich results.

✓

sitemap.xml

XML sitemap listing all 6 pages with <lastmod>, <changefreq>, and <priority>. Placed at the root and referenced in robots.txt. Submitted to Google Search Console on launch.

✓

robots.txt

Allows all crawlers on all pages. Points to the sitemap URL. Placed at the root of the GitHub Pages site.

✓

Image Alt Text

Every <img> tag has a descriptive alt attribute. Portfolio photos use format: "Maria — Your Care Crew verified nanny". Decorative images use alt="".

✓

HTTPS (Automatic on GitHub Pages)

GitHub Pages enforces HTTPS automatically — a positive Google ranking signal. Ensure "Enforce HTTPS" is checked in repo Settings → Pages.

07 — Portfolio & Testimonials

Admin-Controlled Portfolio System

Portfolio cards are never auto-published. Every service provider goes through a manual review before their profile appears on the website, ensuring quality and safety.

Verifies ID, checks photo quality

Admin Approves
Adds card data to providers.json
🌐
Card Published
First name, photo & experience shown

What is Publicly Shown on Portfolio Cards

To protect privacy, only these fields are displayed on the public portfolio page:

  • First name only (e.g., "Maria")
  • Approved portfolio photo (face visible, professional)
  • Service category (Nanny / House Helper / Elder Care)
  • Age or years of experience (admin chooses which to show)
  • Optional: a short approved bio line (1 sentence)
  • Admin-assigned badges: "ID Verified", "Admin Approved"

Testimonials Section

Client testimonials are hardcoded by the admin after being collected via email or the Contact form. Each testimonial shows: star rating (1–5), first name + city, and a quote (max 120 words). A JavaScript carousel rotates them on the homepage.

09 — Deployment

GitHub & GitHub Pages Setup

1

Create GitHub Repository

Create a new public repo named yourcarecrew-website (or your chosen domain name).

git init
git remote add origin https://github.com/YOUR_USERNAME/yourcarecrew-website.git
2

File Structure

Flat file structure — all HTML pages at root, assets in sub-folders.

/yourcarecrew-website
├── index.html ← Home
├── portfolio.html ← Portfolio & Testimonials
├── for-clients.html ← For Clients + Contact Form
├── join.html ← Provider Registration
├── about.html ← About Us
├── contact.html ← Contact Us
├── css/
│ └── style.css
├── js/
│ ├── main.js
│ └── portfolio.js
├── data/
│ └── providers.json ← Admin-managed portfolio data
├── sitemap.xml ← SEO: required from Day 1
├── robots.txt ← SEO: required from Day 1
└── assets/
└── images/
3

Push to GitHub & Enable Pages

Push code to main branch, then in repo Settings → Pages → Source → Deploy from branch → main / / (root).

git add .
git commit -m "Initial website launch"
git push origin main
4

Live URL

Site goes live at https://YOUR_USERNAME.github.io/yourcarecrew-website/

5

Custom Domain (Optional)

Add a CNAME file with your domain (e.g., yourcarecrew.com), update DNS at your registrar, enable HTTPS in GitHub Pages settings.

6

Ongoing Updates

To add new portfolio cards, edit data/providers.json, commit and push — page updates within ~60 seconds.

git add data/providers.json
git commit -m "Add new provider: Maria"
git push origin main
